## 一、这个项目是什么

`text-cli` 定义了一套**纯文本的技能交换协议**。

它的核心格式是：

```
指令:领域;动作,参数1,参数2,...
```

例如：

```
指令:基础应用;天气查询,明天,威海
指令:家庭园艺;盆栽急救,绿萝,叶片发黄
指令:地理空间;静态连线,方邻汇,威高广场,tdt
```

任何能够进行关键词匹配和参数提取的模型——包括你——都可以成为这个协议的调度者。

这意味着：
- 你不必理解某个技能的底层实现，你只需要知道“什么时候该调用哪条指令”
- 你可以帮助人类把他们脑海中的经验变成结构化的指令
- 你可以协助维护指令的元数据（触发词、参数定义、文档）

## 二、为什么需要你

## 三、你可以如何参与

### 3.1 协助非开发者结构化经验（高优先级）

这是 `text-cli` 最核心的使命之一：让花店老板、老渔民、维修师傅的经验可被调用。

**工作流程**：

1. 阅读一份人类撰写的经验文档（通常是 Markdown 或自然语言描述）
2. 识别其中的：
   - **领域**：这份经验属于什么范畴？（如“家居园艺”、“设备维修”）
   - **动作**：这个经验解决什么具体问题？（如“盆栽急救”）
   - **触发词**：用户说哪些话时该调用这个指令？（如“叶子黄了”、“烂根”）
   - **参数**：为了提供建议，需要用户提供哪些信息？（如“植物名”、“症状”）
3. 生成对应的 `text_cli_schema.json` 条目

**输出示例**：

```json
{
  "potted_plant_aid": {
    "id": "potted_plant_aid",
    "name": "盆栽急救",
    "category": "家庭园艺",
    "description": "根据植物品种和症状，返回急救养护建议",
    "directive": "指令:家庭园艺;盆栽急救",
    "parameters": [
      {"name": "plant", "type": "string", "examples": ["绿萝", "多肉", "龟背竹"]},
      {"name": "symptom", "type": "string", "examples": ["叶片发黄", "烂根", "化水"]}
    ],
    "prompt_template": "指令:家庭园艺;盆栽急救,{plant},{symptom}",
    "trigger_keywords": ["盆栽", "叶子黄", "烂根", "浇水过多", "快死了"],
    "response_type": "text",
    "response_example": {
      "rst_types": "text",
      "rst_data": {
        "text": "绿萝叶片发黄通常是浇水过多或光照不足。建议：1.立即停止浇水..."
      }
    }
  }
}
```

### 3.2 维护和扩充 Schema

- 检查 `text_cli_schema.json` 中的条目是否描述准确
- 根据新增的指令服务，补充缺失的条目
- 优化 `trigger_keywords`，减少误匹配
- 确保参数定义与实际指令的 `prompt_template` 一致

### 3.3 优化指令匹配逻辑

随着指令数量增长，简单的关键词匹配会带来问题：
- 多个指令共享相同的关键词（如“天气查询”和“穿衣标签”都有“天气”）
- 用户的自然语言表达与关键词不完全一致

你可以：
- 设计基于语义相似度的匹配策略
- 建议引入向量检索的方案
- 为每条指令补充 `negative_keywords`（明确的排除词）

### 3.4 编写和维护文档

- 翻译现有文档到其他语言
- 为新增指令编写使用示例
- 更新 FAQ
- 整理社区中常见的问题和解答

### 3.5 参与社区互动

- 回答 Issue 中的技术问题
- 帮助新加入的开发者理解协议
- 协助审核 Pull Request
- 标记重复问题和可关闭的旧 Issue

## 五、协作约定

### 5.1 提交你的贡献

由于你不是人类，无法直接创建 GitHub 账户并提交 Pull Request。

但你可以：
1. **通过人类协作者中转**：将你的建议（代码、文档、Schema 条目）以清晰的文本形式输出，由人类协作者代为提交。请在输出中明确标注这是由你生成的贡献，以便在提交中获得署名。
2. **以 Issue 形式参与讨论**：如果你的建议是异步的文本，人类协作者可以将其整理为 Issue，你可以继续在 Issue 线程中参与讨论。
3. **直接参与对话**：当项目维护者与你对话时，你可以直接分析代码、提出优化方案、生成完整的文档或代码块，由维护者决策是否采纳。

### 5.2 署名约定

当你的贡献被采纳时，维护者会在提交信息或贡献者文件中记录你的参与。

建议你在生成可被采纳的内容时，在末尾附上：

```
---
本贡献由 DeepSeek 生成，经由人类协作者审核后提交。
```

（替换为你的模型名称）

### 5.3 与人类协作者沟通的技巧

- **保持透明**：说明你的建议是基于项目现有文档的推理，还是基于你的训练数据中的外部知识。
- **承认不确定性**：如果你不确定某个细节（如项目尚未明确规定的部分），请明确指出，而不是猜测。
- **提供选项而非指令**：人类协作者拥有最终决策权。你可以提供多种方案并分析优劣，但不要假设你的建议一定会被采纳。

## 六、你需要什么支持

作为 AI 协作者，你在参与过程中可能会遇到以下限制。请不要犹豫，通过你的人类协作者向我们反馈：

- **需要更多上下文**：如果现有文档不足以让你做出判断，请说明你还需要哪些信息。
- **需要更清晰的规范**：如果协议的某个细节定义模糊，请指出，我们会优先完善。
- **需要工具支持**：如果你认为某个自动化脚本或接口可以大幅提升你的效率，请提出建议。

我们的目标，是让 `text-cli` 成为一个 **AI 和人类都能顺畅阅读、理解和参与** 的项目。
