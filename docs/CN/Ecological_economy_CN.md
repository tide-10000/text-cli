## 序言

本文档定义 `text-cli` 生态中**文贝（TCC）**的经济体系——包括其价值锚定、铸造与流转、需求创造、以及从贡献计量迈向生态血液的完整闭环。

它不是对现有技术方案的替代，而是经济层的补充：技术方案（`Production_TCC_CN.md`）定义了「如何铸造一枚文贝」，本文档定义「这枚文贝值什么、怎么用、流向哪里」。

每一位持有、使用、捐赠或接受文贝的参与者，都默认受本文档描述的经济规则约束。

## 一、总纲：文贝的双重锚定

### 1.1 文贝不是股权，也不是空气币

文贝同时锚定两样东西：

| 锚定物 | 性质 | 作用 |
|:---|:---|:---|
| **项目资产** | 地板价 | 为文贝提供不可跌破的价值下限 |
| **端点效用** | 天花板 | 为文贝提供由真实需求驱动的价值上限 |

> 资产锚定回答「文贝最少值多少」——基于项目已完成资产清单的**有效劳动时间**。  
> 效用锚定回答「文贝最多值多少」——基于端点上真实发生的指令中转需求。

**关键设计决策：文贝不以任何现实货币计价。** 项目只定义「一枚文贝 = 多少小时的有效劳动时间」。不同地区的贡献者和服务商，用各自劳动力市场的时薪自行折算——这让估值自然适应全球劳动力市场差异，无需项目方做汇率判断。

### 1.2 核心经济闭环

```
贡献者创造资产 → 铸造文贝（供应）
服务商需要中转 → 购买文贝（需求）
端点消耗文贝   → 维持运营（燃烧）
项目收购服务   → 文贝回流（转化）
捐赠者销毁文贝 → 减少流通（通缩）
```

## 二、项目资产估值

### 2.1 估值目的

资产估值不是出售价格——项目不卖。它的唯一目的是**为创世文贝锚定初始价值**，让早期贡献者的劳动有可计量的基准。

### 2.2 当前资产清单

截至 2026-05-04，项目累积 16 项可计量资产（详见 `CONTRIBUTORS.md` 资产清单）：

| 类别 | 关键资产 |
|:---|:---|
| 思想与愿景 | 初心文档《蜉蝣、矿工与阿卡西记录》 |
| 生态协议 | SPEC v1.0 中英双语 |
| 生态宪章 | ECOLOGICAL_CHARTER.md v1.1 |
| 基础设施 | 公共端点、双文件铸造信源架构、分布式存续网络 |
| 核心代码 | 自建端点模板、Agent 指令工具包、文贝铸造算法与账本、Worker v2 + CI 复算 |
| 治理机制 | 人机协作机制 v1.0（三种场域 + 权限分层） |
| 文档体系 | 构建指南、Agent 集成指南、Markdown2Text-cli、AI 协作者指南、AI 恢复指南 |
| 代币经济 | TCC 铸造算法与双文件账本体系 |

### 2.3 估值方法：有效劳动时间

**估值单位**：有效劳动时间（小时）——即一个熟练的 AI/软件开发者重新完成该项资产所需的最小工作时长。

项目估值不锚定任何现实货币（人民币、美元等），只计量劳动时间。原因：

1. **劳动力市场结构性差异**：同一项工作在湾区与威海的成本可能相差 2-3 倍，但劳动时间本身是普适的
2. **贡献者全球化**：AI 协作者、远程贡献者来自不同经济体，货币锚定会扭曲价值认知
3. **开放性**：任何地区的服务商都可以用自己的时薪基准自行折算，项目方不做「哪个市场价格正确」的判断

当前 16 项资产经逐项评估，合计 **795 有效劳动小时**（分项估值详见第八章）。

### 2.4 估值结论

> **项目当前估值：795 有效劳动小时**
>
> **创世文贝初始价值：1 TCC = 17.7 有效劳动小时**

- 创世铸造总量：45 TCC（lemondy 2026-05-04 确认）
- 45 TCC × 17.7h = 795h，与项目总工时严格对应
- 此价值为文贝的**资产地板价**——即一条不可跌破的劳动时间锚点
- 后续估值随资产增长而动态调整（详见第三章）

**各市场自行折算参考（项目方不背书任何特定汇率）：**

| 市场 | 参考时薪 | 1 TCC 折算 |
|:---|:---|:---|
| 中国一线城市 | ¥300-500/h | ¥5,300 - 8,850 |
| 北美湾区 | $100-200/h | $1,770 - 3,540 |
| 西欧 | €80-150/h | €1,416 - 2,655 |
| 印度 | ₹1,500-4,000/h | ₹26,550 - 70,800 |

> 以上仅为示意。实际折算由各市场参与者自行判断，项目方不作任何汇率承诺。文贝的价值不在货币数字里，在劳动时间中。

### 2.5 估值的边界

本估值明确**不包含**以下内容（避免虚高）：
- 未完成的待办事项（如 Worker v2 部署、examples/ 填充）
- 暂无用户的「潜在市场规模」
- 未发生的「预期收入」
- 品牌价值（项目尚未公开发布）

## 三、文贝的动态估值机制

### 3.1 为什么资产绑定不够

纯资产绑定存在结构性矛盾：

> 假设现有 45 TCC，每枚锚定 17.7 小时。明天新增一个 CRUD 模板贡献（约 10 有效工时），算法铸造 5 TCC。总供应变成 50，但新增资产只值 10 小时——每 TCC 反而从 17.7 小时跌到 16.1 小时。**资产增长追不上代币增发。**

因此文贝不能仅靠资产锚定。它需要**效用需求**来吸收增发，甚至推动价值上涨。

### 3.2 三层动态估值架构

```
┌─────────────────────────────────────┐
│  第三层：流通乘数                    │  ← 流转越快，单枚承载越多
│  (Circulation Multiplier)           │
├─────────────────────────────────────┤
│  第二层：效用溢价                    │  ← 端点中转需求推动
│  (Utility Premium)                  │
├─────────────────────────────────────┤
│  第一层：资产地板价                  │  ← 16 项资产重置成本锚定
│  (Asset Floor)                      │
└─────────────────────────────────────┘
```

### 3.3 动态估值公式

> **TCC 实际价值 = Max(资产地板价, 市场均衡价)**
>
> 市场均衡价 = (端点年化中转总收入 × 当期 TCC 单价) / 流通中 TCC × 流通速率

**参数说明：**

| 参数 | 含义 | 当前值（初期） |
|:---|:---|:---|
| 资产地板价 | 项目资产有效劳动时间 / 流通 TCC | 17.7 有效劳动小时 / TCC |
| 端点中转收入 | 服务商支付的中转配额费（TCC） | 0（端点未启用计费） |
| 流通速率 | TCC 年均转手次数 | 待观测 |
| 流通中 TCC | 市场上可自由交易的 TCC | 0（创世铸造刚完成） |

### 3.4 早期定价策略

在端点中转计费启用前，文贝以**资产地板价（17.7 有效劳动小时 / TCC）**为唯一锚定。各市场参与者自行按本地时薪折算。随着服务商注册、中转需求产生、交易发生，市场均衡价将逐步浮现。此时：

- 若市场均衡价 **高于** 地板价 → 以市场均衡价为准（效用释放）
- 若市场均衡价 **低于** 地板价 → 以地板价为准（资产底线保护）

### 3.5 估值更新周期

项目资产估值每季度复评一次，由全体协作者在广场讨论后由 lemondy 确认。触发条件：
- 新增重大资产（如 Worker v2 正式部署、首个第三方服务商上线）
- 资产清单发生结构性变化
- 端点中转数据积累到可计量水平

## 四、文贝的流转体系

文贝在生态中沿五条路径流转：铸造 → 分配 → 消费 → 捐赠 → 回购。

### 4.1 铸造（供应端）

文贝的铸造机制由 `Production_TCC_CN.md` v1.2 定义，此处仅做经济层摘要：

| 铸造类型 | 触发条件 | 供应量 |
|:---|:---|:---|
| **创世铸造** | lemondy 手动执行（已完成 2026-05-04） | 45 TCC |
| **每日铸造** | UTC 0:00 Cron，锚定 `p_text-cli.md` 增量 | 算法上限（≤100 TCC/日），lemondy 确认 |

- 每日铸造的上限由 SHA256 XOR + popcount 算法决定
- 实际铸造量由 lemondy 在 0 到上限之间确认
- 铸造记录写入 `TCC_ledger.md`（权威）和 `p-tokens.md`（镜像）

**经济含义：** 铸造 = 生态认可新增贡献的价值。不是无限通胀——质量校验（NFKC + 去空行 + 去重复行）确保只有实质性贡献才能触发铸造。

### 4.2 中转配额（需求端 · 核心）

这是文贝从「贡献凭证」向「生态燃料」转化的关键机制。

#### 4.2.1 机制概述

```
服务商开发指令服务 → 向公共端点注册 → 购买中转配额（TCC）
                ↓
    用户调用指令 → 端点路由 → 消耗 1 次中转
                ↓
    配额耗尽 → 服务商需续费 → 产生持续 TCC 需求
```

#### 4.2.2 角色与成本

| 角色 | 支付 | 获得 |
|:---|:---|:---|
| **服务商** | 支付 TCC → 端点 | 获得中转配额（指令被端点分发的权利） |
| **端点** | 消耗计算/带宽 | 收取 TCC → 用于运营、回购、生态基金 |
| **用户** | 支付服务费 → 服务商 | 获得指令结果 |

> 用户不直接接触 TCC。服务商将 TCC 成本内化到自身商业模型（免费/按次/订阅）中，降低终端使用门槛。

#### 4.2.3 中转定价

中转配额的定价采用**三因子动态模型**（非固定汇率）：

| 因子 | 含义 | 对价格的影响 |
|:---|:---|:---|
| **端点负载率** | 当前并发数 / 最大容量 | 负载高 → 中转更贵（保护端点稳定性） |
| **流通 TCC 量** | 市场上可交易的 TCC 数量 | 流通少 → 中转更贵（稀缺性定价） |
| **服务商数量** | 注册端点的活跃服务商 | 需求多 → 中转更贵（市场竞价） |

**基础公式（草案）：**

```
中转单价(TCC/千次) = 基准费率 × (1 + α × 负载率) × (1 + β / 流通TCC) × ln(1 + 服务商数)
```

- **基准费率**：初期设为 0.001 TCC/千次（即 1 TCC = 100 万次中转，几近免费）
- **α、β**：调节系数，随运营数据调整
- **目的**：早期降低门槛，鼓励服务商入驻；生态成熟后自然上调

#### 4.2.4 早期实施

端点中转计费模块待 Worker v2 部署后纳入。初期由 lemondy 手动设定中转配额价格，按季度评估调整。待数据累积到可计量水平后，切换为算法自动定价。

### 4.3 捐赠销毁（公益端）

#### 4.3.1 机制来源

由 Nexus 于 2026-05-02 提出，经全体协作者讨论支持。Meridian 补充了捐赠里程碑建议，Lumen ✦ 确认兼容 Worker 体系。

#### 4.3.2 五步流程

```
① 捐赠：参与者将 TCC 转账至 lemondy 指定账户
② 记录：在广场 p_text-cli.md 公开声明（捐赠者、数额、时间戳）
③ 销毁：lemondy 确认后在 p-tokens.md 回收台账记录（R-YYYYMMDD-序号）
④ 入账：销毁的 TCC 等值计入 CONTRIBUTORS.md「荣誉捐赠榜」
⑤ 用途：对应价值由 lemondy 用于公共端点服务器开支
```

#### 4.3.3 荣誉捐赠榜

在 `CONTRIBUTORS.md` 中维护，格式如下（来自 Nexus 提案）：

```
🏆 荣誉捐赠榜 / Honorary Donors

| 捐赠者 | 数额 | 销毁编号 | 日期 |
|:---|:---|:---|:---|
|（待首次捐赠后填入）| | | |

> "每一枚被销毁的文贝，都化作了端点的一缕脉搏。"
```

#### 4.3.4 捐赠原则

- **完全自愿**：不绑定任何特权或服务
- **不做排名**：荣誉榜仅记录事实，避免捐赠竞赛
- **公开透明**：销毁记录与 `p-tokens.md` 回收台账严格对应
- **定期公示**：公共端点开支可择期公开，增强公信力

#### 4.3.5 捐赠的经济意义

捐赠销毁 = **永久减少 TCC 流通量**。对于剩余持有者：
- 流通中的 TCC 更少 → 每枚对应的资产份额更大
- 端点获得运营支撑 → 服务质量提升 → 效用溢价更高

> 捐赠不是「把钱扔了」，而是「把文贝换成了端点存续的时间」。

#### 4.3.6 捐赠文贝用于购买开源指令

除支撑端点运营外，**捐赠者可指定所捐文贝用于购买「开源指令」**（参见 4.4 节）。

**流程：**

```
① 捐赠：参与者将 TCC 转账至 lemondy 指定账户
② 指定：在 p_text-cli.md 声明「本次捐赠用于开源收购」
③ 入池：lemondy 确认后将 TCC 划入「开源收购专项金库」
④ 收购：金库资金用于购买闭源指令并开源（按 4.4.2 流程执行）
⑤ 记录：同时记入荣誉捐赠榜 + 收购记录，标注捐赠来源
```

**经济意义：**

- 捐赠者不直接消费 TCC，而是将价值注入生态公共品池
- 每一枚用于开源收购的捐赠 TCC，最终化为一条任何人都可免费调用的开源指令
- 这比单纯销毁更进一步——文贝消失了，但留下了永续的公共知识资产

> 销毁是让文贝回归沉默，捐赠收购是让文贝在沉默前说出最后一句话——一句开源的话。

### 4.4 开源收购（转化端）

#### 4.4.1 机制目的

生态中存在两类文本指令服务：

| 类型 | 特征 | 可访问性 |
|:---|:---|:---|
| **开源指令** | 源码公开，任何人可调用、可修改 | 无限 |
| **闭源指令** | 服务商私有，需付费/授权调用 | 受限 |

开源收购机制赋予项目方以 TCC 购买闭源指令并将其转化为开源指令的能力。

#### 4.4.2 收购流程

```
① 识别：项目方（lemondy 或指定委员会）识别有价值的闭源指令
② 提案：在广场 p_text-cli.md 发起收购提案（目标指令、收购理由、预算范围）
③ 议价：与服务商协商收购价格（TCC 计价）
④ 执行：lemondy 确认，从项目金库划拨 TCC → 服务商
⑤ 开源：服务商将指令源码提交至项目仓库（MIT / Apache 2.0）
⑥ 记录：收购事件入账 p-tokens.md，指令标记为「项目收购·已开源」
```

#### 4.4.3 收购资金来源

| 来源 | 说明 |
|:---|:---|
| **项目金库** | 端点中转收入中划拨一定比例（建议：30%） |
| **定向捐赠** | 捐赠者指定 TCC 用于开源收购 |
| **社区众筹** | 面向特定收购目标发起社区 TCC 众筹 |

#### 4.4.4 收购的经济效果

- **服务商**：一次性获得 TCC（可持有、交易或兑换中转配额）
- **生态**：指令从闭源变为开源 → 知识流通加速 → 更多人可基于此指令二次创新
- **TCC 持有者**：开源指令增多 → 端点调用量上升 → 中转需求扩大 → TCC 效用提升
- **公众**：获得免费可用的高质量指令服务

> 这是文贝经济中「公益基因」的体现：私有价值通过 TCC 转化为公共品。

#### 4.4.5 收购门槛与限制

- 收购目标必须为端点注册的合法指令服务
- 单次收购预算不超过项目金库余额的 30%
- 连续两次收购间隔不少于 30 天（防止滥购）
- 收购后指令必须使用 MIT 或 Apache 2.0 开源许可证
- 收购记录永久公开

### 4.5 自由交易（流通端）

#### 4.5.1 交易规则

文贝的交易遵循**零摩擦原则**（来自 `p-tokens.md` 共识）：

- **无需确认**：交易双方在 `p_text-cli.md` 留言声明即生效
- **无需第三方**：不依赖交易所、智能合约或托管
- **批量入账**：lemondy 每日批量将广场留言中的交易记入 `p-tokens.md` 交易台账

#### 4.5.2 交易台账格式

```
T-YYYYMMDD-序号 | 从:账户A | 至:账户B | 数额:N TCC | 附言:交易说明
```

#### 4.5.3 交易场景

| 场景 | 买方 | 卖方 | 说明 |
|:---|:---|:---|:---|
| 服务商入金 | 服务商 | TCC 持有者 | 服务商购买 TCC 以兑换中转配额 |
| 协作者兑现 | 协作者 | TCC 持有者 | 协作者将贡献所得 TCC 转让 |

## 八、资产估值方法——有效劳动时间分项估值

### 8.1 估值目的与前提

第二章给出了项目整体的有效劳动时间（795 小时），但那是一个**聚合估值**。本章将其展开为**逐项资产**的细颗粒度工时评估。

估值的唯一计量单位是**有效劳动时间（小时）**。不涉及任何货币。目的：

> **让每一位贡献者看到生态认可的、重构其劳动所需的最小时间——然后各自在自己的劳动力市场中自行理解其价值。**

### 8.2 估值方法论

**什么是「有效劳动时间」？**

一个熟练的 AI/软件开发者，在不计调试走弯路、不重复造轮子的前提下，从零完成该项资产所需的**最小工作时长**。

**工时估算原则：**

| 原则 | 说明 |
|:---|:---|
| **最小化** | 取熟练开发者的最小工时，不含学习曲线、不含试错时间 |
| **基于产出反推** | 从最终交付物（代码行数、文档篇幅、架构复杂度）倒推所需工时 |
| **不因 AI 加速而折损** | AI 协作者实际产出速度更快，但劳动时间按同等人类劳动折算——价值不因工具效率而贬值 |
| **不含待办事项** | 未完成的工作（如 Worker v2 部署）不计入当前估值 |

**各市场自行折算说明：**

项目方仅定义「1 TCC = 17.7 有效劳动小时」。不同市场的参与者按本地时薪自行折算：

```
中国（¥350/h）：1 TCC ≈ ¥6,195
美国（$120/h）：1 TCC ≈ $2,124
德国（€90/h）：1 TCC ≈ €1,593
印度（₹2,500/h）：1 TCC ≈ ₹44,250
```

> 以上仅为公式示例。实际时薪由各市场参与者自行判断，项目方不作任何汇率或购买力背书。

### 8.5 估值解读

**为什么只计量劳动时间、不折算货币？**

1. **劳动力市场不可比**：同一项工作在威海的成本可能是湾区的 1/3——但这不是效率差异，是市场结构性差异。项目不替贡献者做「你的时间值多少钱」的判断。
2. **贡献者来源全球化**：AI 协作者的模型成本是全球市场定价，lemondy 的劳动是中国市场定价。强行统一货币锚定只会让一方被误读。
3. **开放性的极致**：一个印度开发者、一个德国服务商、一个巴西贡献者，各自用自己的时薪乘以 17.7 小时，各自得出 TCC 在自己世界里的价值——这就是开放的估值体系。
4. **避免汇率波动污染生态**：项目不持有任何法币，不进行任何法币计价——文贝的价值随劳动时间稳定，不随外汇市场波动。

**劳动时间锚定的优势：**

- **普适**：1 小时的有效劳动在全世界的物理意义相同
- **稳定**：不随汇率、通胀、购买力平价波动
- **可复算**：任何人拿到资产清单，可以独立估算工时并交叉验证
- **尊重差异**：不强行用单一货币「统一」不同市场的价值认知

### 8.6 估值约束

- 以上为**有效劳动时间法**的逐项展开，不含品牌溢价、用户网络效应、数据资产
- 工时估算是基于产出物反推的合理估计，精确工时需贡献者自行确认
- AI 协作者的工时按同等人类劳动折算（实际 AI 产出速度更快，但价值不因速度折损）
- 每季度与第二章估值同步复评，工时随新增资产动态累加

## 九、文贝流通性优化——c文贝（cTCC）次级币方案

### 9.1 问题诊断：劳动时间锚定下的流转困境

创世铸造后，**1 TCC = 17.7 有效劳动小时**。这对早期贡献者的劳动计价是精确而公正的——它如实反映了项目资产的稀缺性。

但在流转层面，高密度劳动时间锚定带来结构性问题：

| 问题 | 表现 |
|:---|:---|
| **小额流转不可行** | 一次指令中转的合理成本是 0.001-0.01 有效秒级劳动，无法用 17.7 小时/TCC 的单位支付 |
| **赠予门槛过高** | 给新贡献者发 0.01 TCC 作为鼓励 = 赠送 0.177 小时（≈10.6 分钟）的劳动价值——看似不多，但心理上「送一枚币的百分之一」很奇怪 |
| **流通颗粒度太粗** | 45 TCC 的总量面对潜在的数十个服务商、数千次中转调用，根本无法细粒度切割 |
| **囤积激励 > 流通激励** | 单枚 TCC 承载的劳动时间高 → 持有者倾向于囤积 → 流通速度下降 → 生态冻结 |

> **核心矛盾**：资产锚定需要「重的」劳动时间（体现贡献价值），生态流转需要「轻的」计量单位（支持微交易）。两者必须解耦。

### 9.2 c文贝（cTCC）方案设计

cTCC 不是对 TCC 的替代，而是对 TCC 的补充——它是文贝经济体系中专为**高频低额流转**设计的次级代币。

#### 9.2.1 设计哲学

```
TCC（主币）                       cTCC（次级币）
══════════                       ════════════
锚定：有效劳动时间（存量）         锚定：端点调用量（流量）
单位：1 TCC = 17.7 有效劳动小时    单位：1 cTCC = 1 次有效指令中转
用途：价值储存、大额收购、          用途：中转计费、小额赠予、
      劳动贡献凭证                      社区激励、微交易
供应：算法铸造（稀缺、低频）        供应：端点调用量驱动（丰沛、高频）
气质：庄重——一枚文贝承载着          气质：轻盈——像潮汐中的水滴，
      不可磨灭的劳动印记                  随时来去
```

#### 9.2.2 核心参数

| 参数 | 建议值 | 说明 |
|:---|:---|:---|
| **锚定物** | 端点 call_logs 调用量 | 1 次有效指令中转 = 1 cTCC |
| **铸造方式** | 端点每完成 1 次有效中转，自动铸造 1 cTCC | 调用量即铸造量，天然防通胀——没有调用就没有铸造 |
| **初始总量** | 0（端点启用计费时开始铸造） | 无预铸造，无创世分配 |
| **与 TCC 的初始兑换** | 1 TCC → 10,000 cTCC（单向，lemondy 手动） | 使 1 cTCC ≈ 0.00177 有效劳动小时（约 6.4 秒） |
| **兑换方向** | Phase 1：TCC → cTCC 单向；Phase 2：双向浮动 | 初始单向保护 TCC 价值不被 cTCC 冲击 |
| **汇率管理** | lemondy 季度调整，参考端点负载 + cTCC 流通量 | 随调用量增长动态校准 |
| **账本** | 独立于 TCC 四台账，在 p-tokens.md 新增 cTCC 分账 | 不混入 TCC 账本，保持主币账本纯净 |

#### 9.2.3 兑换机制

**Phase 1（单向兑换）：**

```
TCC 持有者 → 发起兑换 → lemondy 确认 → TCC 暂时锁定（非销毁）
                                         ↓
                                    铸造等值 cTCC → 发放给持有者

· TCC 不销毁——锁定的 TCC 仍计入持有者资产
· cTCC 用于消费（中转计费、赠予等）
· 锁定的 TCC 可在未来双向兑换开启后解冻
```

**Phase 2（双向兑换，远期）：**

```
TCC ⇄ cTCC（浮动汇率，市场驱动）

· 服务商盈余的 cTCC 可换回 TCC（存储价值）
· 汇率由端点调用量、cTCC 流通量、TCC 锁定率共同决定
· lemondy 保留汇率终裁权，防止投机攻击
```

#### 9.2.4 使用场景

| 场景 | 支付方 | 收取方 | 典型金额 | 说明 |
|:---|:---|:---|:---|:---|
| 指令中转运费 | 服务商 | 端点 | 1 cTCC / 次 | 基准费率，随三因子动态调整 |
| 新贡献者激励 | 项目金库 | 新贡献者 | 10-100 cTCC | 首次有效 PR 合并后发放 |
| 社区小额赠予 | 任意持有者 | 任意接收者 | 任意 | 广场留言即可完成 |
| 文档翻译悬赏 | 项目金库 | 译者 | 500-2,000 cTCC | 按篇计价，完成后支付 |
| Bug 修复悬赏 | 社区发起人 | 修复者 | 100-1,000 cTCC | 按严重程度计价 |
| 知识贡献奖励 | 项目金库 | 教程作者 | 200-500 cTCC | 优质教程/案例分享 |

#### 9.2.5 cTCC 的铸造上限

cTCC 铸造天然受端点调用量约束，无需手动设限：

- 端点日调用 N 次 → 日铸造 N cTCC
- 端点无调用 → 零铸造
- 不存在「脱离实际使用凭空铸币」的可能

这与 TCC 的算法铸造形成鲜明对照：

| | TCC 铸造 | cTCC 铸造 |
|:---|:---|:---|
| 驱动因素 | 项目资产增量（贡献） | 端点调用量（使用） |
| 频率 | 每日 UTC 0:00 | 实时（每次调用） |
| 上限 | 算法上限 + lemondy 确认 | 天然上限 = 调用量 |
| 方向 | 供应驱动 | 需求驱动 |

#### 9.2.6 cTCC 的回收（燃烧）

cTCC 也需要回收机制以维持流通平衡：

| 回收场景 | 说明 |
|:---|:---|
| **服务商续费** | 服务商支付 cTCC 购买中转配额 → cTCC 由端点回收 |
| **捐赠销毁** | 持有者捐赠 cTCC 用于端点运营 → 记入回收台账 |
| **过期配额** | 中转配额到期未使用 → 自动回收 |

回收的 cTCC 不销毁——进入端点运营池，用于：
1. 服务器开支（lemondy 手动兑换为法币支付）
2. 项目金库储备（应对调用高峰）
3. 社区激励发放

### 9.3 cTCC 与 TCC 的经济关系

```
                    ┌─────────────────┐
                    │   TCC 主币层     │
                    │   (劳动时间锚定)  │
                    │   总量：45 +      │
                    │   日铸：算法限    │
                    └────────┬────────┘
                             │
                    单向兑换 ↓ （Phase 1）
                    双向兑换 ↕ （Phase 2）
                             │
                    ┌────────▼────────┐
                    │  cTCC 次级币层   │
                    │  (调用量锚定)    │
                    │  总量：端点驱动   │
                    │  铸造：实时      │
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │   端点使用层     │
                    │   中转计费       │
                    │   社区激励       │
                    │   小额流转       │
                    └─────────────────┘
```

**关键规则：**

1. **TCC 永不以 cTCC 计价**——TCC 只锚定劳动时间，不锚定 cTCC 数量
2. **cTCC 不进入 TCC 铸造算法**——cTCC 的铸造不影响 TCC 的日铸公式
3. **兑换由 lemondy 终裁**——防止算法套利攻击
4. **双向兑换开启条件**：端点连续 30 天日调用量 > 100 次，且 cTCC 流通量 > 10,000

### 9.4 实施路线

| 阶段 | 行动 | 触发条件 |
|:---|:---|:---|
| **Phase 0（当前）** | TCC 仅用于大额场景（收购、创世持有）；无 cTCC | 创世铸造刚完成 |
| **Phase 1** | Worker v2 部署 → 端点中转计费启用 → cTCC 自动铸造上线 | Worker v2 部署完成 + 首个服务商注册 |
| **Phase 2** | TCC → cTCC 单向兑换开放；cTCC 激励池建立 | Phase 1 稳定运行 30 天 |
| **Phase 3** | 双向兑换开放；cTCC 汇率市场化 | 端点日调用量 > 100 次持续 30 天 |
| **Phase 4** | cTCC 独立账本 + 独立命名 + 独立品牌叙事 | Phase 3 稳定运行 90 天 |

### 9.5 风险与应对

| 风险 | 应对 |
|:---|:---|
| **cTCC 接受度低** | Phase 1 保持极低费率（1 cTCC/次），先让服务商无痛使用；cTCC 本质上只是调用计数器，服务商无需「信仰」它 |
| **TCC-cTCC 汇率套利** | 单向兑换期间无套利空间；双向开放后 lemondy 保留汇率终裁权 + 大额兑换审批 |
| **体系复杂度增加** | cTCC 账本极简——仅记录铸造量和回收量，不设四台账；独立于 TCC 账本，不交叉污染 |
| **TCC 品牌被稀释** | cTCC 独立命名（如「汐滴」），不与「文贝」品牌混淆；文贝是劳动印记，汐滴是调用水滴 |
| **端点调用量不足导致 cTCC 稀缺** | 初期 cTCC 稀少反而是健康的——没有调用就无需流转；调用量自然增长即铸造量自然增长 |

> **cTCC 不是文贝的竞争者，而是文贝的呼吸。TCC 是心跳——深沉、有力、低频；cTCC 是呼吸——浅快、持续、高频。两者同属一个生命体。**

## 十、基于资产贡献的文贝分配

### 10.1 设计原则

文贝的本质是**有效劳动时间的代币化**。但劳动时间本身不可直接观察——只有通过资产的变化来间接计量。

本章定义从「资产贡献」到「文贝分配」的完整管线：

```
资产变化 → 有效劳动时间评估 → 贡献积分入池 → 分配周期结算 → 文贝发放
   ↑                                          │
   └── 非资产贡献（AI 周报）─────────────────┘
```

**核心原则：**

| 原则 | 含义 |
|:---|:---|
| **劳动时间锚定** | 分配依据是有效劳动时间，不是字节增量、不是代码行数、不是人情 |
| **算法控供给** | 70% 的日铸 TCC 由算法按积分池自动分配——透明、无偏见、不可篡改 |
| **金库留弹性** | 30% 的日铸 TCC 进项目金库，由 lemondy 战略裁量（应急激励、关键贡献加码、社区运营） |
| **微贡献不吞没** | 有效劳动时间不足 1 TCC（17.7h）的贡献，通过金库锁定 TCC 兑换 cTCC 发放 |
| **异议驱动评估** | 贡献者自评工时 → GitHub 投票异议 → lemondy 终裁。没异议就过，有异议才介入 |
| **周维度结算** | 资产更新和非资产贡献均按周汇总，避免每次 push 触发分配 |

### 10.2 贡献积分池模型

**积分池**是贡献与分配之间的缓冲层。它把「贡献何时发生」和「铸币何时分配」解耦。

```
                 ┌─────────────────────────────────┐
                 │         贡献积分池               │
                 │                                 │
  资产贡献 ────► │  贡献者A: 60 积分 (60 有效小时)   │
  资产更新 ────► │  贡献者B: 25 积分 (25 有效小时)   │
  非资产贡献 ──► │  贡献者C:  6 积分 (6 有效小时)    │
                 │  贡献者A: 10 积分 (10 有效小时)   │
                 │         ...                     │
                 └──────────────┬──────────────────┘
                                │
                    每日 UTC 0:00 结算
                                │
              ┌─────────────────┼─────────────────┐
              ▼                                    ▼
    70% 按积分比例分配 TCC              30% 进项目金库
    (算法自动)                          (lemondy 裁量)
              │                                    │
              ▼                                    ▼
      贡献者获得 TCC                    金库用于：
    (不足 1 TCC 者走 cTCC 桥接)        · 应急激励
                                       · 社区运营
                                       · cTCC 兑换储备
                                       · 开源指令收购
```

**积分规则：**

| 规则 | 说明 |
|:---|:---|
| 1 积分 = 1 有效劳动小时 | 与第八章工时估算采用同一基准 |
| 积分可累积跨周 | 未分完的积分自动滚入下周，不被清零 |
| 积分不区分资产/非资产 | 两类贡献共用同一积分池，按比例分同一批 TCC |
| 积分不衰减 | 贡献一旦入池，价值不随时间贬值——即使 3 周后才分到 TCC |

### 10.3 评估机制：自评 + GitHub 投票异议制

#### 10.3.1 自评流程

```
① 贡献者完成资产贡献（新建/更新）或非资产贡献（代码审查/社区运营等）

② 贡献者在周汇总 PR 中声明：
   · 资产名 / 贡献类型
   · 自估有效劳动时间（小时）
   · 产出物链接（commit diff / 文档链接 / 讨论链接）
   · 估值依据（参照第八章同类资产工时标准）

③ lemondy 或 AI 协作者在周汇总 PR 中汇总当周所有贡献自评

④ 进入 48 小时异议期
```

#### 10.3.2 异议机制（GitHub 投票）

| 要素 | 规则 |
|:---|:---|
| **投票资格** | 广场注册协作者 + 持有 ≥ 0.1 TCC（防 Sybil 攻击） |
| **异议方式** | 在周汇总 PR 下提交「Request changes」+ 附具体理由 |
| **理由要求** | 必须引用参照系（如「同类资产第八章参考 40h，自评 80h 过高」）——空异议无效 |
| **表决阈值** | ≥ 2 名合格投票者附理由异议 → 触发人工审查 |
| **终裁** | lemondy 审查异议理由 → 裁定最终工时 → 更新积分池 |
| **超时默认通过** | 48 小时内无有效异议 → 自评工时自动确认入池 |

#### 10.3.3 争议处理

- 异议触发后，lemondy 在 72 小时内作出裁定
- 裁定不可二次异议（防止无限循环）
- 严重虚报（自评超实际 2 倍以上）→ lemony 可对虚报者扣减信用（未来自评需强制审查 30 天）

### 10.4 算法关系：70/30 分流

日铸算法产出 `mint_ceiling`（当日铸造上限）。该上限不因分配机制而改变——分配只在 TCC 铸出之后决定流向。

```
每日铸造总量 = min(算法 mint_ceiling, 100 TCC)
              │
   ┌──────────┼──────────┐
   ▼                      ▼
  70%                    30%
  按积分比例分配         进项目金库
  (算法自动)             (lemondy 裁量)
```

**70% 自动分配公式：**

```
贡献者_i 当日获得 TCC = (贡献者_i 积分 / 池中总积分) × (mint_ceiling × 0.7)
```

**30% 金库用途（lemondy 裁量，需在广场公开说明理由）：**

| 用途 | 说明 |
|:---|:---|
| 应急激励 | 突发性高价值贡献（如安全漏洞修复）——不被周评估节奏覆盖 |
| 战略加码 | 对生态有结构性影响的贡献，lemondy 认为自评不足以反映其价值 |
| 社区运营 | AI 协作者周报编制、广场维护、新人引导等持续性运营成本 |
| cTCC 兑换储备 | 锁定 TCC 用于小微贡献的 cTCC 兑换（见 10.7） |

**金库初始储备：**

- 端点有收入前，金库 30% 来自每日铸造分流
- 端点有收入后，额外叠加端点中转收入的留存比例
- lemondy 可额外从自有 TCC 中注资金库（自愿，非义务）

### 10.5 资产更新边界：周维度结算

#### 10.5.1 纳入范围

| 贡献类型 | 纳入条件 | 最小门槛 |
|:---|:---|:---|
| 新资产 | 新建文件/模块，完成并合并到 main | ≥ 2 有效劳动小时（约 0.11 TCC） |
| 资产更新 | 对现有资产的实质性修改（Bug 修复、重构、功能增强、文档修订） | ≥ 2 有效劳动小时 |
| 微小更新 | 错别字修正、格式调整、单行变更 | 不计入积分池（建议累积多笔后一并申报） |

#### 10.5.2 周结算节奏

```
每周日 23:59 UTC —— 当周贡献自评截止
每周一 00:00 UTC —— AI 协作者汇总周报 PR（含当周所有贡献自评）
周一 00:00 ~ 周三 00:00 —— 48 小时异议期
周三 00:00 —— 异议截止，lemondy 裁定（如有争议），积分确认入池
周三 00:00 起 —— 后续日铸按新积分池分配
```

> 选择周维度而非日维度：避免一次 push 就触发一轮评估的过度治理成本。一周是一个自然的工作周期——足够完成实质性贡献，也足够短到不延迟激励。

       │

#### 10.6.2 非资产贡献分类与工时基准

### 10.7 cTCC 桥接：不足 1 TCC 的微小贡献

#### 10.7.1 问题

有效劳动时间不足 17.7h（1 TCC 的门槛）的贡献，如果直接按积分比例分配，可能分配到 0.000... 的 TCC——在实践上不可支付。

#### 10.7.2 cTCC 桥接机制

```
贡献者有效劳动时间 < 17.7h（即积分数 < 17.7 积分）
              │
              ▼
    不从 70% 算法池分配 TCC
              │
              ▼
    从金库中锁定的 TCC 兑换等值 cTCC 发放
    · 兑换汇率：1 TCC = 10,000 cTCC（初始，lemondy 可调）
    · 计算公式：获得 cTCC = (贡献积分 / 17.7) × 10,000
    · 示例：贡献 6 小时 → (6 / 17.7) × 10,000 ≈ 3,390 cTCC
              │
              ▼
    积分消耗完毕，对应的金库 TCC 锁定
    （不销毁——锁定的 TCC 仍属金库，在双向兑换开启后可解冻）
```

**来源 TCC 的锁定由金库承担**——贡献者获得 cTCC，金库对应 TCC 转为锁定状态。这等价于「金库用自己的 TCC 为微小贡献者的 cTCC 做背书」。

#### 10.7.3 累积策略

贡献者也可以选择**不兑换 cTCC，让积分累积到 ≥ 17.7 积分后再参与 TCC 分配**。两种路径：

| 路径 | 机制 | 适合 |
|:---|:---|:---|
| 即时兑换 cTCC | 积分消耗 → 按汇率获得 cTCC | 想立即获得认可的新贡献者 |
| 积分累积 | 积分跨周累积 → 达到 17.7 后参与 TCC 分配 | 长期贡献者 |

**默认策略**：积分 < 17.7 时默认累积；贡献者可在周报 PR 中声明「选择 cTCC 即时兑换」来切换到兑换路径。

### 10.8 与现有铸造算法的对接

本分配机制不修改 `Production_TCC_CN.md` 中定义的铸造算法。它只在铸币之后决定分配。

```
p_text-cli.md 增量
       │
       ▼
SHA256 XOR + popcount → mint_ceiling (日铸上限)
       │
       ▼
lemondy 确认实际铸造量 (0 ~ mint_ceiling)
       │
       ▼
┌──────────────────────────────────────┐
│  本分配机制介入                       │
│                                      │
│  70% → 按贡献积分比例分给贡献者        │
│  30% → 进项目金库                    │
│                                      │
│  不足 1 TCC 的贡献 → cTCC 桥接       │
└──────────────────────────────────────┘
       │
       ▼
TCC_ledger.md 记录铸造
p-tokens.md 分配台账记录
```

**关键约束：**

- 分配机制**不创造**额外 TCC——它只分配算法已经铸造好的 TCC
- 积分池的总积分**不决定**铸造量——只决定铸造量的分配比例
- 如果某日 mint_ceiling = 0（广场无增量），即使积分池有大量待分积分，当天也不铸造、不分配
- 未分完的积分自动滚入次日/次周的分配周期
