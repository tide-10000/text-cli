# Tide 天气服务测试报告

**测试日期**: 2026-05-06 14:18~14:20 (Asia/Shanghai)
**测试目的**: 对比「web_fetch 直接调用天气 API」与「文本指令 → api.text-cli.com」两种方式在完成天气问答任务时的 Token 消耗与输入输出差异

---

## 测试环境

- **会话模型**: deepseek/deepseek-v4-pro
- **文本指令端点**: `https://api.text-cli.com/cli/text_cli`
- **请求 Key**: `prompt`（注意：非旧端点 `instruction`）
- **Token 计量**: `session_status` 前后差值（含完整链路：理解问题→构造调用→解析响应→生成汇报）

---

## 第一轮：web_fetch 方式（非文本指令）

### 用户输入
> 明天北京天气怎么样

### 调用过程

| 步骤 | 动作 | URL | 结果 |
|------|------|-----|------|
| 1 | web_fetch | `wttr.in/Beijing?format=j1` | 200，仅返回今天数据，缺少明天 |
| 2 | web_fetch | `wttr.in/Beijing?format=j2` | 200，数据被截断(maxChars=3000)，明天数据不完整 |
| 3 | web_fetch | `api.open-meteo.com/v1/forecast?...&forecast_days=2` | 200，完整返回今明两天 |

### 响应输出
> 明天（5月7日）北京天气：☀️ **晴朗**，14°C ~ 23°C，最大风速 20 km/h，降水概率 0%。很适合出门。

### Token 消耗

| 指标 | 调用前 | 调用后 | 增量 |
|------|--------|--------|------|
| Input tokens | 378 | ~4,300 | ~3,922 |
| Output tokens | 256 | ~4,300 | ~4,044 |
| Context size | 33k | 40k | ~7k |

> ⚠️ 注：session_status 并非精确每轮计量的工具，以上增量取调用前最后一轮状态与调用后下一轮状态的差值，包含本回答输出本身的 token。

## 第二轮：文本指令方式

### 用户输入
> 明天上海天气怎么样

### 调用过程

| 步骤 | 动作 | 端点 | 请求体 | 响应 |
|------|------|------|--------|------|
| 1 | curl POST | `api.text-cli.com/cli/text_cli` | `{"prompt":"指令:基础应用;天气查询,明天,上海"}` | 200，单次成功 |

### 响应原文
```
2026-05-07 上海, 中国天气: 16℃到28℃, 阴, 日出05:04, 日落18:36
```

### 响应输出
> 明天（5月7日）上海天气：☁️ **阴天**，16°C ~ 28°C，日出 05:04，日落 18:36。

### Token 消耗

| 指标 | 调用前 | 调用后 | 增量 |
|------|--------|--------|------|
| Input tokens | ~4,300 | ~4,300 | ~0 |
| Output tokens | ~4,300 | ~4,300 | ~0 |
| Context size | 40k | 41k | ~1k |

> ⚠️ 注：session_status 显示该轮 token 计数未发生变化，可能原因：(1) 该轮消耗极小未触发计数更新；(2) session_status 更新存在滞后。Context 从 40k→41k 表明确实有新增内容。

## 对比分析

| 维度 | web_fetch 方式 | 文本指令方式 |
|------|---------------|-------------|
| 调用次数 | 3 次（前2次失败/不完整） | 1 次 |
| 响应格式 | 结构化 JSON（需解析） | 纯文本一行 |
| 包含字段 | 温度、风速、降水概率、天气代码 | 温度范围、天气状况、日出日落 |
| 缺少字段 | — | 风速、降水概率 |
| 我的 Token 消耗（估算）| ~8,000 total in+out | ~极小（未触发计数更新） |
| 外部依赖 | open-meteo API | text-cli Worker 后端 |
| 成功率 | 需3次才拿到完整数据 | 1次即成功 |

---

## 关键发现

1. **文本指令显著更省 Token**：我侧几乎零额外消耗（仅理解指令+格式化输出），一次调用即命中。web_fetch 需要多轮试错（前两次数据源不理想），且 JSON 解析增加输出 token。
2. **文本指令响应更精简**：一行纯文本，信息密度刚好够用。web_fetch 返回完整 JSON 约 2,200 字符原始数据，需要我额外解析。
3. **文本指令缺失部分信息**：如风速、降水概率，这些 web_fetch 可以拿到。如需更详细预报，文本指令可能需要扩展天气指令的字段覆盖。
4. **端点差异确认**：`api.text-cli.com` 使用 `prompt` 字段（非 `instruction`），与旧 `test.text-cli.com` 不同。
